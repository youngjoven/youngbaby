## 1. iOS 로컬 데이터 스키마 (SwiftData)

> iOS 17 이상에서 동작하는 SwiftData를 사용한다.

### UserProfile
| 필드 | 설명 |
|---|---|
| id | 로컬 고유 식별자 |
| babyName | 아이 이름 |
| babyBirthDate | 아이 생년월일 (월령 자동 계산 기준) |
| motherName | 어머니 이름 |
| cognitoUserId | Cognito 연동 사용자 ID |

### FeedingRecord
| 필드 | 설명 |
|---|---|
| id | 로컬 고유 식별자 |
| feedingTime | 수유 시간 (버튼 클릭 시 현재 시각 자동 저장) |
| amountMl | 분유량 (ml, 사용자 직접 입력) |
| syncedAt | 클라우드 동기화 시각 (nil이면 미동기화) |

### BowelRecord
| 필드 | 설명 |
|---|---|
| id | 로컬 고유 식별자 |
| bowelTime | 배변 시간 (기록 버튼 클릭 시 현재 시각 자동 저장) |
| condition | 배변 상태: `hard` / `normal` / `soft` |
| syncedAt | 클라우드 동기화 시각 (nil이면 미동기화, 백엔드 배포 전 UI에서 미표시) |

## 2. AWS 클라우드 백엔드 아키텍처

### 전체 구조
```
[iOS App]
    │
    ├─ Cognito (인증)
    │
    ├─ API Gateway → Lambda
    │       ├─ /feedings    (수유 기록 CRUD)
    │       ├─ /bowels      (배변 기록 CRUD)
    │       ├─ /profile     (프로필 조회/수정)
    │       ├─ /advisor     (어드바이저 추천 요청)
    │       └─ /insights    (인사이트 조회)
    │
    ├─ DynamoDB (데이터 저장)
    │
    ├─ SNS → APNs (수유 알람)
    │
    └─ Amazon Bedrock (AI Agent)
```

### DynamoDB 테이블 설계

**feeding-records 테이블**
| 키 | 값 |
|---|---|
| Partition Key | `userId` |
| Sort Key | `feedingTime` (ISO 8601) |
| 속성 | `amountMl`, `createdAt` |

**bowel-records 테이블**
| 키 | 값 |
|---|---|
| Partition Key | `userId` |
| Sort Key | `bowelTime` (ISO 8601) |
| 속성 | `condition`, `createdAt` |

**user-profiles 테이블**
| 키 | 값 |
|---|---|
| Partition Key | `userId` |
| 속성 | `babyName`, `babyBirthDate`, `motherName` |

**insights 테이블**
| 키 | 값 |
|---|---|
| Partition Key | `userId` |
| Sort Key | `generatedAt` (ISO 8601) |
| 속성 | `insightType`, `content` |

### REST API 설계

| Method | Path | 설명 |
|---|---|---|
| GET | /profile | 프로필 조회 |
| PUT | /profile | 프로필 수정 |
| POST | /feedings | 수유 기록 생성 |
| GET | /feedings | 수유 기록 목록 조회 |
| DELETE | /feedings/{id} | 수유 기록 삭제 |
| POST | /bowels | 배변 기록 생성 |
| GET | /bowels | 배변 기록 목록 조회 |
| DELETE | /bowels/{id} | 배변 기록 삭제 |
| POST | /advisor/advice | 어드바이저 추천 요청 |
| GET | /insights | 인사이트 목록 조회 |

### Cognito 가족 공유 계정 구조

- Cognito User Pool에서 **이메일 + 비밀번호** 기반 로그인
- 엄마/아빠가 **동일한 계정(이메일/비밀번호)** 으로 각자 기기에서 로그인
- 로그인한 계정의 `userId`를 기준으로 DynamoDB 데이터를 공유
- 별도 초대/연동 기능 없이 **계정 공유 방식**으로 운영

## 3. 로컬 ↔ 클라우드 동기화 전략

**방식: 오프라인 우선 (Offline-First)**

1. 기록 생성 시 **로컬(SwiftData)에 먼저 저장**
2. 네트워크 연결 상태를 확인 후 클라우드로 동기화
3. 동기화 완료 시 `syncedAt` 필드를 현재 시각으로 업데이트
4. 앱 실행 시 미동기화(`syncedAt == nil`) 항목을 일괄 업로드
5. 클라우드 최신 데이터를 로컬로 다운로드하여 병합

**충돌 처리:** 동일 `userId` 기준, `createdAt` 최신 항목을 우선

## 4. Notification (수유 알람) 시스템 흐름

**알람 방식:** SNS/APNs 서버 알람 (다중 기기 동시 알람)

```
수유 기록 완료
    │
    ▼
현재 월령 계산
    │
    ▼
월령별 권장 수유 간격 조회
(0~1개월: 2~3h / 1~3개월: 3~4h / 3~6개월: 3~4h / 6~12개월: 4~5h)
    │
    ▼
다음 알람 시각 = 수유 시간 + 권장 간격 중간값
    │
    ▼
Lambda → SNS → APNs → 엄마·아빠 기기 동시 알람 발송
    │
    ▼
알람 수신 → 사용자 인지 → 수유 진행 → 기록
```

**알람 내용:** "수유할 시간입니다 🍼 분유 수유 후 기록해 주세요."

## 5. 스마트 수유 어드바이저 Agent 아키텍처

**기술:** Amazon Bedrock (Claude 3.5 Sonnet)

### API 흐름
```
[iOS App] → POST /advisor/advice
    │
    ▼
Lambda
    ├─ DynamoDB에서 최근 7일 수유 기록 조회
    ├─ DynamoDB에서 최근 7일 배변 기록 조회
    ├─ 현재 월령 계산
    └─ 월령별 권장 기준 로드
    │
    ▼
Bedrock API 호출 (프롬프트 생성 후 전달)
    │
    ▼
응답 파싱 → JSON 반환
    │
    ▼
[iOS App] 추천 카드 UI 표시
```

### 프롬프트 구조
```
당신은 영아 분유 수유 전문 어드바이저입니다.
아래 데이터를 분석하여 한국어로 간결한 추천을 제공하세요.

[아이 정보]
- 현재 월령: {ageMonths}개월
- 월령별 권장 수유 간격: {recommendedInterval}
- 월령별 1회 권장 분유량: {recommendedAmount}ml

[최근 7일 수유 기록]
{feedingRecords}

[최근 7일 배변 기록]
{bowelRecords}

아래 3가지 항목을 각각 1~2문장으로 작성하세요:
1. 다음 수유 시간 조정 추천
2. 분유량 조언
3. 수유-배변 종합 의견

면책: 이 추천은 참고용이며 의료적 진단을 대체하지 않습니다.
```

### 응답 포맷 (JSON)
```json
{
  "nextFeedingAdvice": "다음 수유는 3시간 후를 추천합니다.",
  "amountAdvice": "현재 월령 기준 1회 140ml를 권장합니다.",
  "overallOpinion": "이번 주 배변 상태와 수유량을 보면 전반적으로 양호합니다.",
  "disclaimer": "이 추천은 참고용이며 의료적 진단을 대체하지 않습니다."
}
```

### 전달 방식
- **즉시 추천 카드**: 수유 기록 저장 직후 화면 하단에 카드 자동 표시
- **어드바이저 탭**: 탭 진입 시 최신 추천 결과 조회 (캐시 활용)

## 6. 데이터 인사이트 Agent 아키텍처

**기술:** Amazon Bedrock
**전달 방식:** 앱 내 인사이트 카드/피드 (Push Notification 미사용)

### 분석 트리거 조건
- 최소 3일 이상 기록 축적 시 분석 가능
- **앱 실행 시** 마지막 분석으로부터 7일 경과 여부 확인 → 경과 시 Lambda 트리거

### 인사이트 생성 흐름
```
앱 실행
    │
    ▼
마지막 인사이트 생성 시각 확인
    │
    ├─ 7일 미경과 → 기존 인사이트 카드 표시
    │
    └─ 7일 경과 (또는 최초)
            │
            ▼
        GET /insights → Lambda 트리거
            │
            ▼
        DynamoDB에서 최근 4주 수유·배변 기록 조회
            │
            ▼
        Bedrock API 호출 (인사이트 프롬프트)
            │
            ▼
        인사이트 생성 → DynamoDB insights 테이블 저장
            │
            ▼
        앱 내 인사이트 카드/피드 표시
```

### 인사이트 유형 및 조건
| 유형 | 감지 조건 |
|---|---|
| 분유량 변화 감지 | 이번 주 평균 분유량이 지난주 대비 ±15% 이상 변화 |
| 수유 간격 변화 | 평균 수유 간격이 30분 이상 변화 |
| 배변 이상 패턴 | 최근 3일간 묽음 또는 단단함이 연속 발생 |
| 월령별 권장량 비교 | 하루 평균 분유량이 권장량 대비 ±20% 이상 차이 |
| 수유-배변 연관 | 분유량과 배변 횟수의 상관관계 패턴 감지 |

## 7. 앱 주요 화면 구성 (UI/UX)

### 탭 구조 (하단 탭바, 4개 탭)

```
┌─────────────────────────────┐
│         홈 (Home)            │  ← 오늘 요약 + 즉시 수유 기록 버튼
├─────────────────────────────┤
│       기록 (Record)          │  ← 수유/배변 기록 목록 + 통계
├─────────────────────────────┤
│   어드바이저 (Advisor)        │  ← AI 추천 카드 조회
├─────────────────────────────┤
│   인사이트 (Insights)         │  ← 주간 인사이트 카드/피드
└─────────────────────────────┘
```

### 화면별 주요 구성

**홈 탭**
- 아이 이름 + 현재 월령 표시
- 다음 권장 수유 시각
- **[수유 기록하기] 버튼** → 클릭 시 분유량 슬라이더 모달(50~300ml, 10단위) → 저장 후 어드바이저 추천 카드 표시
- **[배변 기록하기] 버튼** → 배변 상태 선택 모달 (시간은 현재 시각 자동 기록)
- 오늘 수유 횟수 / 오늘 총 분유량 / 다음 수유 예정 시각 요약

**기록 탭**
- 수유/배변 탭 전환 (Segmented Picker)
- 수유 간격 평균 / 1회 평균 분유량 / 하루 총 분유량 통계 카드
- 각 기록 행에 🗑️ 삭제 버튼 표시 (실수 기록 즉시 삭제 가능)

**어드바이저 탭**
- 최신 AI 추천 카드 (다음 수유 시간 / 분유량 조언 / 수유-배변 종합 의견)
- 백엔드 미배포 시 빈 상태 메시지 표시
- 면책 문구 표시 필수

**인사이트 탭**
- 주간 인사이트 카드 피드 (유형별 카드)
- 백엔드 미배포 시 빈 상태 메시지 표시
- 면책 문구 표시

### 온보딩 화면 (최초 실행)
1. 아이 이름 입력
2. 아이 생년월일 입력 (→ 월령 자동 계산)
3. 어머니 이름 입력
4. 시작하기 → SwiftData 로컬 저장 + 알림 권한 요청
> ⚠️ 현재 구현: Cognito 연동 미완료. 로컬 프로필만 저장됨. 백엔드 배포 후 Cognito 회원가입/로그인 추가 예정.

### iOS 구현 세부 사항 (현재 완료)
| 항목 | 세부 내용 |
|---|---|
| 색상 에셋 | `Assets.xcassets`: PastelBackground(#FFF8F0), PastelPink(#FFB3C6), PastelMint(#B3F0E0), PastelPurple(#D4B3FF) |
| 텍스트 색상 | 파스텔 색상은 배경 전용. 텍스트는 진한 버전 RGB 값 직접 사용 |
| 다크 모드 | `RootView().preferredColorScheme(.light)` 으로 라이트 모드 고정 |
| 분유량 입력 | Slider (50~300ml, 10단위), 큰 숫자 실시간 표시 |
| 배변 시간 | `Date()` 자동 기록, UI에서 입력 제거 |
| 동기화 아이콘 | 백엔드 배포 전까지 `syncedAt` 상태 아이콘 UI에서 숨김 |
| 앱 아이콘 | "뀨" 문자 기반 아이콘 (크림색 배경 + 핑크 원 + 흰 텍스트) |
| 네비게이션 타이틀 | `.principal` ToolbarItem + `.inline` 모드로 탭별 이모지+색상 타이틀 표시 |
| 배변 요약 표시 | 홈 화면 배변 타일 서브타이틀: "🟡 최근 묽음" 형태로 마지막 배변 상태 표시 |
| 뷰 상태 구조 | `ZStack { if isLoading { ProgressView } else if empty { emptyState } else { ScrollView { content } } }` 패턴 일관 적용 |

## 질의응답

**[Question] 1:** Amazon Bedrock에서 사용할 AI 모델을 선택해야 합니다. 어떤 모델을 사용하시겠습니까?
1) Claude 3.5 Sonnet (고성능, 높은 정확도)
2) Claude 3 Haiku (빠른 응답, 낮은 비용)
3) Amazon Nova Pro (AWS 자체 모델)
**[Answer]** Claude 3.5 Sonnet을 사용하겠습니다.

**[Question] 2:** 알람 방식을 확정해야 합니다. 초기 버전에서 어떤 방식으로 구현하시겠습니까?
1) iOS 로컬 알람만 (인터넷 없이도 동작, 단일 기기)
2) SNS/APNs 서버 알람 (다중 기기 동시 알람, 인터넷 필요)
3) 두 가지 모두 (로컬 알람 기본, 다중 기기 시 SNS 보완)
**[Answer]** AWS 서버에서 알람을 발송하겠습니다. 따라서 SNS/APNs 서버 알람을 사용하겠습니다. 
**APNs는 외부 서버 -> IPhone 알람을 가능하게 해주는 Apple의 중계 서버입니다.(참고차 적음)**

**[Question] 3:** iOS 최소 지원 버전을 결정해야 합니다. SwiftData는 iOS 17 이상에서만 동작합니다.
1) iOS 17 이상 (SwiftData 사용 가능, 최신 기기 중심)
2) iOS 16 이하도 지원 (Core Data 사용, 더 넓은 호환성)
**[Answer]** iOS 17 이상을 사용합니다.

**[Question] 4:** 앱 UI 디자인 컨셉을 결정해야 합니다.
1) 파스텔 톤 + 귀여운 아이콘 (아기자기한 느낌)
2) 화이트 + 민트/하늘색 계열 (깔끔하고 의료적인 느낌)
3) 다크 모드 지원 포함 여부도 결정 필요
**[Answer]** 파스텔 톤 + 귀여운 아이콘의 아기자기한 느낌을 사용합니다.
