# 기획 (Inception): 영아 분유 수유 관리 애플리케이션
**대상:** 생후 0개월~12개월 영아 / 분유 수유 전용 (모유 수유 미포함)
**Role:** AWS Solution Architect

## User Stories (사용자 스토리)
1. **[알람 수신 및 확인]** 사용자는 정해진 시간에 앱을 통해 알람(Notification)을 수신하고, 이를 인지하여 아이에게 제때 수유할 수 있어야 한다.
2. **[분유 수유 기록]** 사용자는 실제 분유 수유를 마친 후, 버튼을 클릭하면 수유 시간(현재 시각)이 자동으로 기록되며, 분유량(ml)은 별도로 입력한다.
3. **[데이터 자동 계산]** 사용자는 앱이 기록된 데이터를 기반으로 '수유 간격 평균', '1회 평균 분유량', '하루 총 분유량'을 자동으로 계산하여 보여주기를 원한다.
4. **[패턴 분석 시각화]** 사용자는 일정 기간 동안의 수유 패턴과 변화를 한눈에 파악할 수 있는 통계/요약 정보를 제공 받기를 원한다.
5. **[스마트 수유 어드바이저]** 사용자는 아이의 수유 기록과 월령 정보를 기반으로, AI Agent가 다음 수유 시간 조정, 분유량 조언 등 맥락에 맞는 맞춤형 추천을 받기를 원한다.
6. **[데이터 인사이트]** 사용자는 AI Agent가 수유 데이터를 능동적으로 분석하여 "이번 주 분유량이 지난주 대비 15% 감소했습니다" 같은 변화 감지, 이상 패턴 알림, 월령별 권장량 비교 등의 인사이트를 자동으로 제공받기를 원한다.
7. **[배변 활동 기록]** 사용자는 아이의 배변이 발생할 때마다 배변 시간과 배변 상태(단단함/정상/묽음)를 앱에 직접 기록할 수 있어야 한다. 하루 배변 횟수는 기록을 바탕으로 앱이 자동 집계한다.
8. **[수유-배변 연관 분석]** 사용자는 분유 수유 데이터와 배변 활동 데이터를 종합하여, 아이가 잘 먹고 잘 배변하고 있는지에 대한 분석 결과를 제공받기를 원한다.

## 시스템 개발 계획 (워크플로우 단계별 체크리스트)

### 0. 기획 (Inception)
- [x] 사용자 스토리 정의 (User Stories 1~8)
- [x] 질의응답을 통한 요구사항 명확화 (Question 1~6)

### 1. 도메인 설계 (Domain Design)
- [ ] 핵심 도메인 모델(Entity) 정의:
  - `FeedingRecord` (수유 시간, 분유량)
  - `BowelRecord` (배변 시간, 배변 상태: 단단함/정상/묽음)
  - `UserProfile` (아이 이름·생년월일 + 어머니 이름 포함, 단일 아이 전제, 대상 월령 0~12개월)
    - 월령은 생년월일과 현재 날짜를 기반으로 앱이 자동 계산 (사용자가 직접 입력 불필요)
- [ ] 핵심 비즈니스 로직(Domain Service) 설계: 수유 간격 계산, 총 분유량 집계, 하루 배변 횟수 자동 집계, 수유-배변 연관 분석
- [ ] 알람 예측 로직 설계: 월령별 권장 수유 간격(대한소아과학회·WHO 기준) 기반 자동 조정
  - 0~1개월: 2~3시간 / 1~3개월: 3~4시간 / 3~6개월: 3~4시간 / 6~12개월: 4~5시간
- [ ] 스마트 수유 어드바이저 Agent 도메인 설계: 입력(수유 기록, 아이 월령) → 출력(수유 시간 조정 추천, 분유량 조언) 정의
- [ ] 데이터 인사이트 Agent 도메인 설계: 수유 변화 감지 규칙, 배변 이상 패턴 기준, 수유-배변 연관 분석 규칙, 월령별 권장량 기준 데이터 정의
- [ ] 월령별 권장 분유량·수유 간격 데이터 확보 전략 정의: 대한소아과학회·WHO 공개 가이드라인 중 저작권 안전한 데이터만 활용 (생후 0~12개월 기준), 앱 내 "참고용" 면책 문구 명시, 저작권 보호 자료는 절대 사용하지 않음

### 2. 논리적 설계 (Logical Design)
- [ ] iOS 로컬 데이터 스키마 설계 (SwiftData/Core Data)
- [ ] AWS 클라우드 백엔드 아키텍처 설계 (다중 기기 연동)
  - [ ] API Gateway + Lambda (REST API)
  - [ ] DynamoDB (수유 기록 및 배변 기록 저장)
  - [ ] Cognito (사용자 인증): 가족 공유 계정 구조 설계 (엄마/아빠가 동일 계정으로 로그인 → 같은 아이 데이터 공유)
  - [ ] SNS/Push Notification 연동 설계 (수유 알람 전용: 앱이 수유 시간을 예측하여 사용자에게 "수유할 시간입니다" 알람 발송)
  - [ ] Amazon Bedrock 연동 설계 (스마트 수유 어드바이저 Agent)
- [ ] 로컬 ↔ 클라우드 데이터 동기화 전략 설계
- [ ] Notification 시스템 흐름 및 동작 방식 아키텍처 설계
- [ ] 스마트 수유 어드바이저 Agent 아키텍처 설계: 프롬프트 설계, API 흐름, 응답 포맷
- [ ] 데이터 인사이트 Agent 아키텍처 설계: 분석 트리거 조건, 인사이트 생성 주기, 앱 내 인사이트 카드 표시 방식 (Push Notification 미사용)
- [ ] 앱 주요 화면 구성 및 사용자 흐름(UI/UX) 설계

## 질의응답 (Clarification Questions)

**[Question] 1:** 수유 기록 등 앱의 데이터는 iOS 기기 내부(예: Core Data, SwiftData)에만 오프라인으로 저장 되나요, 아니면 다중 기기 연동이나 향후 확장을 위해 **AWS 클라우드 백엔드**를 연동하는 아키텍처로 설계해야 하나요?
**[Answer]** 다중 기기 연동이나 향후 확장을 위해 AWS 클라우드 백엔드를 연동하는 아키텍처로 설계합니다.

**[Question] 2:** 알람(Notification)은 어떤 방식으로 예약(Trigger)되나요? 
1) 사용자가 직접 "다음 수유는 언제"라고 수동으로 타이머나 알람을 맞추는 방식
2) 사용자의 최근 기록을 기반으로 앱이 적절한 시간을 예측하여 자동으로 알람을 추천/설정해 주는 방식
**[Answer]** 사용자의 최근 기록을 기반으로 앱이 적절한 시간을 예측하여 자동으로 알람을 추천/설정해 주는 방식으로 진행합니다. 시간은 스마트폰의 시간을 기준으로 잡습니다. 알람 간격은 대한소아과학회·WHO 기준 월령별 권장 수유 간격을 기반으로 자동 조정됩니다. (0~1개월: 2~3시간 / 1~3개월: 3~4시간 / 3~6개월: 3~4시간 / 6~12개월: 4~5시간)

**[Question] 3:** 스마트 수유 어드바이저 Agent의 추천은 어떤 형태로 사용자에게 전달되나요?
1) 수유 기록 직후 즉시 추천 카드를 보여주는 방식 (예: "다음 수유는 3시간 후, 140ml 추천")
2) 별도 "어드바이저" 탭/화면에서 사용자가 직접 조회하는 방식
3) 두 가지 모두
**[Answer]** 두 가지 모두 넣었으면 좋겠습니다. 

**[Question] 4:** 데이터 인사이트 Agent의 인사이트 전달 방식은 어떻게 하시겠습니까?
1) 앱 내 인사이트 카드/피드로 표시 (앱 열었을 때 확인)
2) 중요한 변화 감지 시 Push Notification으로 능동적 알림
3) 두 가지 모두
**[Answer]** 앱 내의 인사이트 카드/피드로 표시해 주세요.

**[Question] 5:** 월령별 권장 수유량 기준 데이터는 어떤 출처를 사용할까요? 대한소아과학회, WHO 등 공신력 있는 기관의 가이드라인을 기반으로 하되, 앱 내에서 "참고용"임을 명시하는 방향이 적절할까요?
**[Answer]** 대한소아과학회, WHO 기관의 가이드라인을 기반으로 해 주시고, 앱 내에서 참고용임을 명시하는 방향이 적절해 보입니다. 저작권에 위배될 행동을 절대로 하지 않습니다. 저작권이 있으면 절대로 참고하지도, 게시하지도 않습니다.

**[Question] 6:** 앱은 단일 아이를 대상으로 하나요, 아니면 다자녀(여러 아이)를 지원해야 하나요? `Baby` 엔티티 및 로그인/회원가입 시 입력 항목 설계에 직접적인 영향을 줍니다.
**[Answer]** 단일 아이로 구성합니다. 아이의 이름과 어머니의 이름을 입력하도록 하며, 이 정보는 추후 로그인/회원가입 시 활용됩니다.
