## iOS 신생아 수유 관리 애플리케이션 기획 개요

목표
분유 수유 중인 영아(생후 0개월~12개월)를 위한 iOS 애플리케이션을 개발하고, 최종적으로 Apple App Store에 배포(Publish)하는 것을 목표로 합니다. (모유 수유는 포함하지 않으며, 분유 수유만을 대상으로 합니다.)

핵심 기능
이 애플리케이션에서 가장 중요한 기능은 알람(Notification) 입니다.
iPhone (예: iPhone 15 Pro)에서 알람이 울리면 사용자는 이를 인지하고, 실제로 아이에게 분유를 먹이는 행동을 하게 됩니다. (대상: 생후 0~12개월 영아, 분유 수유 전용)

즉, 알람 → 사용자 인지 → 실제 수유 행동 → 앱에 기록의 흐름을 기반으로 동작하는 애플리케이션입니다.

## 주요 데이터 기록 기능

알람이 발생한 시간을 기준으로, 사용자는 다음 정보를 애플리케이션에 기록합니다.

[분유 수유 기록]
수유 시간: 버튼 클릭 시 현재 시각이 자동으로 기록됨 (사용자 직접 입력 불필요)
분유량: 사용자가 몇 ml를 먹였는지 별도 입력

[배변 활동 기록] 사용자가 직접 입력
배변 시간 (몇 시 몇 분에 배변했는지)
배변 상태: 단단함(변이 굳어 있음) / 정상(너무 단단하지도 묽지도 않음) / 묽음(물기가 많음)
※ 하루 배변 횟수는 개별 기록을 바탕으로 앱이 자동 집계

## 데이터 분석 및 계산 기능

애플리케이션은 기록된 데이터를 기반으로 다음과 같은 정보를 자동으로 계산해야 합니다.

[분유 수유 분석]
수유 시간 기록: 각 수유가 이루어진 정확한 시간
수유 간격: 이전 수유와 다음 수유 사이의 시간 간격
월령별 권장 수유 간격과 비교 (대한소아과학회·WHO 기준)
- 0~1개월: 2~3시간마다 / 1~3개월: 3~4시간마다 / 3~6개월: 3~4시간마다 / 6~12개월: 4~5시간마다
1회 평균 분유량: 한 번 수유할 때 평균적으로 몇 ml를 먹는지
하루 총 분유량: 하루 동안 아이가 섭취한 분유의 총량
수유 패턴 분석: 일정 기간 동안의 수유 패턴 및 변화

[배변 활동 분석]
하루 배변 횟수: 개별 기록을 자동 집계
배변 상태 추이: 단단함/정상/묽음 빈도 변화
수유-배변 연관 분석: 분유량·수유 간격과 배변 상태의 상관관계

## 핵심 시스템 흐름

알람(Notification) 생성 (월령별 권장 수유 간격 기반 자동 조정)
사용자가 알람을 확인
실제 수유 진행
앱에 수유 시간 및 분유량 기록
(별도) 배변 발생 시 사용자가 배변 시간 및 상태 직접 기록
앱이 데이터를 저장하고 자동 계산 수행
통계 및 요약 정보 제공 (수유 + 배변 종합)

## AI Agent 기능

### 스마트 수유 어드바이저
아이의 수유 기록과 월령 정보를 기반으로, AI Agent가 다음 수유 시간 조정, 분유량 조언 등 맥락에 맞는 맞춤형 추천을 제공합니다.
- 수유 기록 직후 즉시 추천 카드 표시
- 별도 어드바이저 탭에서 사용자가 직접 조회 가능

### 데이터 인사이트
AI Agent가 수유 데이터를 능동적으로 분석하여 변화 감지, 이상 패턴 알림, 월령별 권장량 비교 등의 인사이트를 자동으로 제공합니다.
- 앱 내 인사이트 카드/피드로 표시

## 워크플로우 설계
1. 기본은 계획 모드

사소하지 않은 모든 작업은 계획 모드로 시작 (3단계 이상이거나 구조적 결정이 필요한 경우)
뭔가 잘못되면 즉시 멈추고 다시 계획 세우기 — 그냥 밀어붙이지 말 것
계획 모드는 개발할 때만이 아니라 검증 단계에서도 활용
모호함을 줄이려면 처음부터 상세 스펙을 작성할 것

2. 서브에이전트 활용

메인 컨텍스트 창을 깔끔하게 유지하려면 서브에이전트를 적극적으로 활용
조사, 탐색, 병렬 분석은 서브에이전트에 맡길 것
복잡한 문제일수록 서브에이전트를 더 많이 투입
서브에이전트 하나당 하나의 작업만 — 집중 실행을 위해

3. 자기개선 루프
사용자에게 수정을 받을 때마다: 해당 패턴을 'plan.md'에 기록
같은 실수를 반복하지 않도록 스스로 규칙을 작성
실수율이 낮아질 때까지 이 교훈들을 반복해서 다듬을 것
세션 시작 시 해당 프로젝트의 교훈 목록을 먼저 검토

4. 완료 전 반드시 검증
작동한다는 걸 증명하기 전까지 절대 작업 완료로 표시하지 말 것
필요한 경우 main과 변경 사항 간의 동작 차이를 비교
스스로에게 물어볼 것: "시니어 엔지니어가 이걸 승인할까?"
테스트 실행, 로그 확인, 정확성 입증

5. 우아함을 추구할 것 (균형 있게)
사소하지 않은 변경이라면: 잠깐 멈추고 "더 우아한 방법이 있지 않을까?" 자문
수정이 어설프게 느껴진다면: "지금 내가 아는 모든 걸 바탕으로, 우아한 해결책을 구현해"
단순하고 명백한 수정엔 이 과정을 생략 — 오버엔지니어링 금지
결과물을 내놓기 전에 스스로 검토

6. 자율적인 버그 수정
버그 리포트를 받으면: 그냥 고칠 것. 하나하나 물어보지 말 것
로그, 에러, 실패한 테스트를 직접 찾아서 해결
사용자가 컨텍스트를 전환할 필요가 없도록
어떻게 하라는 말 없이도 실패한 CI 테스트를 직접 가서 고칠 것

## 작업 관리
**계획 먼저**: 체크 가능한 항목으로 'task.md'에 계획 작성
**계획 검토**: 구현 시작 전에 확인
**진행 상황 추적**: 진행하면서 완료 항목 표시
**변경 사항 설명**: 각 단계마다 상위 수준 요약 제공
**결과 문서화**: 'task.md'에 검토 섹션 추가
**교훈 기록**: 수정 후 'task.md' 업데이트

## 핵심 원칙
**단순함 우선**: 모든 변경은 최대한 단순하게. 건드리는 코드는 최소화.
**게으름 금지**: 근본 원인을 찾을 것. 임시방편 없음. 시니어 개발자 기준으로.
**최소 영향**: 꼭 필요한 부분만 수정. 새로운 버그를 만들지 말 것.
